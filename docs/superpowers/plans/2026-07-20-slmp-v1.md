# Mitsubishi SLMP / MC Protocol (device side) v1 Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Host SLMP (MELSEC Communication), 3E binary frame over TCP, in-process so a SCADA MC-protocol driver can read and write the app's tags by **device code + address** (`D0`, `M100`).

**Architecture:** Pure codec layers (`slmp_frame` → `slmp_commands` → `slmp_device_image`) under `protocols/slmp/`, a `SlmpMap` binding named tags to device/address, a **length-prefixed TCP** host (mirroring S7/EtherNet-IP), and the program's fourth use of the Python (`pymcprotocol`) probe lane. Mirrors the FINS/S7 device-image + map + dispatch stack, reusing the shared write-gate. Two deliberate contrasts from FINS: **TCP** (not UDP) and **LITTLE-endian** (not big-endian).

**Tech Stack:** Flutter/Dart + a `pymcprotocol` (pure-Python) E2E probe. `flutter test`, `flutter analyze`, `flutter build web --release`, `tool/slmp_e2e.sh`.

## Global Constraints

- **ADR-010**: pure Dart, in-process, no companion process, no FFI.
- Pure Dart (no Flutter, no `dart:io`) in `mobile/lib/protocols/slmp/` and `mobile/lib/models/`; `dart:io` confined to `mobile/lib/services/slmp_host.dart`.
- **Codecs must NEVER throw** on malformed/truncated/hostile input.
- Deterministic: no wall clock, no randomness in codec logic.
- Additive/backward-compatible: a project with no `slmp` key loads disabled at the default port; no existing serialized form or scan behaviour changes; default-projects round-trip and scan-equivalence stay green.
- **Force-aware writes**: reuse `isExternallyWritable` (`models/tag_write_gate.dart:53`) so a write to a forced / read-only / reserved-`System` tag is refused, tag unchanged.
- Dark theme; `withValues(alpha:)` never `withOpacity`; braces on all control flow; zero `flutter analyze` warnings; no overflow at 320/360/1400.
- No competitor-tooling branding: no "OpenPLC"/"Beremiz"/"CODESYS"/"RSLogix"/"GX Works"/"MELSOFT". `SLMP`/`MELSEC` are protocol names and are used freely (as `Modbus`/`DNP3`/`S7comm`/`FINS` already are); a controller family (`Q`/`L`/`iQ-R`) only where a technical distinction requires it. No reverse-engineering wording.

> ### LITTLE-ENDIAN — READ FIRST, THE HEADLINE TRAP
> SLMP binary is **LITTLE-ENDIAN throughout** — subheader, length fields, device number, and all word data. This is the **exact inverse** of S7 and FINS, whose big-endian codecs sit right next door in `protocols/s7/` and `protocols/fins/`. **Do not copy an `Endian.big` from a neighbouring area protocol.** Every codec function states its endianness in its doc comment, and every test asserts byte order **against hand-built little-endian buffers with literal expected bytes** — a build→parse round-trip cancels an endianness error and proves nothing. FINS's frame file (`fins_frame.dart:7-12`) carries exactly this warning naming EtherNet/IP; the SLMP files must carry it naming S7/FINS.

> ### THE 32-BIT WORD ORDER — the one genuine unknown, DO NOT ASSUME
> A 32-bit value (`DINT`/`REAL`) spans **two consecutive words**; which word holds the high half is a documented Mitsubishi gotcha. FINS flagged the identical thing as provisional and its **real client OVERTURNED the guess**. So: pick a provisional word order, **isolate it to two encode/decode helpers** (mirror FINS's `_reverseWordOrder` at `fins_area_image.dart:87-98`), pin it with a **literal-byte test**, and let the Task 5 `pymcprotocol` E2E settle it. **The E2E must use the independent-seed technique** (Task 5), NOT a symmetric write→read-back — a round-trip is byte-transparent through symmetric encode/decode and cannot detect a word swap. Do NOT copy FINS's settled low-word-first order blind; Mitsubishi's convention is its own and the client decides.

> ### WIRE-DETAIL CAVEAT (every task)
> The real `pymcprotocol` client (Tasks 3, 5) is the authority. Implement what the brief says; if the client disagrees, **the client is right** — fix the code and report it. That is why the probe runs early.

### Task 1: SLMP 3E binary frame

**Files:** Create `mobile/lib/protocols/slmp/slmp_frame.dart`, test `mobile/test/slmp_frame_test.dart`.

**Interfaces:** `SlmpHeader { int network; int pc; int destModuleIo; int destModuleStation; int monitoringTimer; }`, `SlmpFrame { SlmpHeader header; int command; int subcommand; Uint8List data; }`, `SlmpFrame? parseSlmpRequest(Uint8List)`, `Uint8List buildSlmpResponse({required SlmpHeader requestHeader, required int endCode, Uint8List data})`, plus `const int kSlmpRequestSubheader = 0x5000;`, `const int kSlmpResponseSubheader = 0xD000;`, end-code constants.

**Context (all LITTLE-ENDIAN):**
- **Request**: `subheader` u16 (`0x5000`), `network` u8, `pc` u8 (`0xFF`), `destModuleIo` u16 (`0x03FF`), `destModuleStation` u8, `requestDataLength` u16 (counts the bytes that FOLLOW this field: monitoring timer + command + subcommand + data), `monitoringTimer` u16, `command` u16, `subcommand` u16, data.
- **Response**: `subheader` u16 (`0xD000`), the echoed routing (`network`, `pc`, `destModuleIo`, `destModuleStation`), `responseDataLength` u16 (counts end code + data), `endCode` u16 (`0x0000` = success), data.
- **Routing accepted permissively**: network/PC/module echoed back unvalidated (the same call as S7 rack/slot).
- End codes: `kSlmpEndNormal = 0x0000`, `kSlmpEndCommandError = 0xC059`, `kSlmpEndAddressRange = 0xC056`, `kSlmpEndPointCount = 0xC051`. (Confirm the exact values against the client at Task 3; these are the error-path set.)

- [ ] **Step 1: Write the failing tests** (`slmp_frame_test.dart`):
  - `parseSlmpRequest` on a hand-built little-endian request returns the exact fields; assert `command` from two DIFFERENT bytes (e.g. bytes `0x01, 0x04` → `0x0401` little-endian, NOT `0x0104`), so a big-endian read FAILS.
  - `buildSlmpResponse` emits `0xD000` little-endian, the echoed routing, a correct `responseDataLength` (= 2 for the end code + data.length), the `endCode` little-endian, then data — assert the literal bytes.
  - `parseSlmpRequest` returns `null` for a buffer too short to hold the fixed header + command/subcommand (no throw).

### Task 2: Batch Read/Write + device codes

**Files:** Create `mobile/lib/protocols/slmp/slmp_commands.dart`, test `mobile/test/slmp_commands_test.dart`.

**Interfaces:** `SlmpDeviceSpec { int deviceCode; int deviceNumber; int pointCount; }`, `SlmpDeviceSpec? parseBatchReadRequest(Uint8List commandData)`, `({SlmpDeviceSpec spec, Uint8List writeData})? parseBatchWriteRequest(Uint8List commandData)`, `Uint8List buildBatchReadResponseData(Uint8List words)`, command/subcommand + device-code constants.

**Context (LITTLE-ENDIAN):**
- **Batch Read (word)** = command `0x0401`, subcommand `0x0000`; **Batch Write (word)** = command `0x1401`, subcommand `0x0000`; **bit units** = subcommand `0x0001`.
- **Device spec (binary 3E)**: a **3-byte device number (LITTLE-ENDIAN)** + a **1-byte device code**, then `pointCount` u16. (Total device-spec header = 6 bytes.)
- **Device codes (v1)**: `kSlmpDevD = 0xA8`, `kSlmpDevM = 0x90`, `kSlmpDevW = 0xB4`, `kSlmpDevR = 0xAF`. (Confirm the exact byte values against `pymcprotocol`'s `DeviceConstants` at Task 3.)
- **Read response** = end code `0x0000` + the requested words (little-endian). **Write** command data = device spec + write words; response = end code only (no data).
- **Write-length validation**: a Batch Write declaring N word points must carry exactly `2*N` trailing bytes — reject a mismatch with `null` BEFORE any slice (mirror FINS's `parseMemAreaWriteItem` bounds check; a parser that slices `2*N` without checking is an out-of-bounds crash vector, and the host feeds this arbitrary frame bytes).
- Parse functions return `null` on a short/malformed command (never throw).

- [ ] **Step 1: Write the failing tests** (`slmp_commands_test.dart`):
  - `parseBatchReadRequest` on `[deviceNumber 3 bytes LE, deviceCode, pointCount u16 LE]` returns the exact fields; assert the 3-byte device number from differing bytes (e.g. `0x64,0x00,0x00` → 100, and a case like `0x00,0x01,0x00` → 256, NOT a big-endian misread).
  - `parseBatchWriteRequest` on a spec + 2 write words → the spec plus the exact write bytes; a write declaring 2 points but carrying 1 word returns `null` (bounds check), traced to fail BEFORE the slice.
  - `buildBatchReadResponseData` round-trips known little-endian words (assert literal bytes).
  - Each parse returns `null` on a truncated command.
