# Siemens S7comm (device side) v1 Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Host S7comm on TCP 102 in-process so a SCADA S7 driver can read and write the app's tags by **memory area + byte offset**.

**Architecture:** Pure codec layers (`tpkt_cotp` → `s7_pdu` → `s7_area_image`) under `protocols/s7/`, an `S7Map` binding named tags to area/offset, a length-prefixed socket host (the only `dart:io` file), an Outbound Protocols card, and a **real `python-snap7` client proving the connect handshake at Task 3 — before any read/write logic exists.**

**Tech Stack:** Flutter/Dart + a `python-snap7` E2E probe reusing WS2's Python lane. `flutter test`, `flutter analyze`, `flutter build web --release`, `tool/s7_e2e.sh`.

## Global Constraints

- **ADR-010**: in-process, pure Dart, no companion process, no FFI.
- Pure Dart (no Flutter, no `dart:io`) in `mobile/lib/protocols/s7/` and `mobile/lib/models/`; `dart:io` confined to `mobile/lib/services/s7_host.dart`.
- **Codecs must NEVER throw** on malformed/truncated/hostile input — return `null` or an error PDU. The host feeds them raw wire bytes.
- Deterministic: no wall clock, no randomness in codec logic.
- Additive/backward-compatible: a project with no `s7comm` key loads disabled at port 102; no existing serialized form or scan behaviour changes; default-projects round-trip and scan-equivalence stay green.
- **Force-aware writes**: a write landing on a *forced* or `ReadOnly` tag is refused, tag unchanged, checked against the **ROOT** tag so a member path cannot bypass it.
- Dark theme; `withValues(alpha:)` never `withOpacity`; braces on all control flow; zero `flutter analyze` warnings; no overflow at 320/360/1400.
- No competitor-tooling branding: no "OpenPLC"/"Beremiz"/"CODESYS"/"RSLogix"/"SIMATIC"/"STEP 7"/"TIA Portal". `S7comm` is a protocol name and is used freely (as `Modbus`/`DNP3`/`OPC UA` already are); naming a controller family (`S7-300`, `S7-1500`) is acceptable only where a technical distinction requires it, never as a compatibility claim. No reverse-engineering wording.

> ### ENDIANNESS — READ THIS FIRST, IT IS THE MOST LIKELY SOURCE OF A WIRE BUG
> **S7comm is BIG-ENDIAN throughout**: the TPKT length, every multi-byte S7 header and parameter field, and all encoded values. The immediately preceding workstream (EtherNet/IP) was **little-endian everywhere**, and its code is right next door in `protocols/enip/`. Do not copy an endianness from there.
> Every codec function must **state its endianness in its doc comment**, and every test must assert byte order **against hand-built buffers with literal expected bytes** — a pure build→parse round-trip cancels an endianness error and proves nothing.

> ### WIRE-DETAIL CAVEAT (applies to every task)
> S7comm is a large, partly community-documented protocol. The constants below make the work concrete, but **the real `python-snap7` client is the authority.** If the client disagrees with this plan, **the client is right** — fix the code and report the deviation prominently. That discovery is why Task 3 exists.

### Task 1: TPKT + COTP framing

**Files:**
- Create: `mobile/lib/protocols/s7/tpkt_cotp.dart`
- Test: `mobile/test/tpkt_cotp_test.dart`

**Interfaces:**
- Produces: `const int kTpktHeaderLen = 4;`, `TpktHeader? parseTpkt(Uint8List)` (exposing `version`, `length`), `Uint8List buildTpkt(Uint8List payload)`, `CotpPacket? parseCotp(Uint8List)` (exposing `pduType`, `payload`, and for CR/CC the `srcRef`/`dstRef`/`srcTsap`/`dstTsap`), `Uint8List buildCotpConnectConfirm({required int srcRef, required int dstRef, required int srcTsap, required int dstTsap})`, `Uint8List buildCotpData(Uint8List payload)`, plus PDU-type constants.

**Context:**
- **TPKT (RFC 1006)**: `version 0x03`, `reserved 0x00`, `length` u16 **BIG-ENDIAN** counting the **whole packet including the 4-byte TPKT header**. (Contrast: EtherNet/IP's length excluded its header. Getting this wrong shifts every frame boundary.)
- **COTP (ISO 8073)**: first byte is the **length indicator** — the header length **excluding the LI byte itself**. Then the PDU type.
  - **CR** `0xE0` / **CC** `0xD0`: dest ref u16, src ref u16, class/option u8, then variable parameters, each `code u8, len u8, value`: `0xC1` source TSAP, `0xC2` destination TSAP, `0xC0` TPDU size.
  - **DT** `0xF0`: LI `0x02`, then the TPDU-number/EOT byte (`0x80` = last data unit).
- The destination TSAP encodes rack and slot. v1 **accepts any rack/slot** and echoes the client's TSAPs back in the CC — this is a simulator, and rejecting a mismatch gives a confusing failure with no diagnostic value.

- [ ] **Step 1: Write the failing tests**

Create `mobile/test/tpkt_cotp_test.dart` covering:
- `parseTpkt` on a hand-built buffer `[0x03, 0x00, 0x00, 0x16, ...]` returns `version == 3` and `length == 0x0016` (22) — **asserting big-endian**; a test with a length whose two bytes differ (e.g. `0x01, 0x02` → 258, not 513) so a little-endian implementation FAILS.
- `buildTpkt` emits `0x03, 0x00`, then the big-endian **total** length (payload length **+ 4**), then the payload — assert the literal bytes.
- `parseTpkt` returns `null` for a buffer shorter than 4 bytes (no throw).
- `parseCotp` on a hand-built **CR** extracts pduType `0xE0`, both refs, and both TSAP parameter values, regardless of parameter order.
- `buildCotpConnectConfirm` emits pduType `0xD0` with a correct length indicator, and round-trips through `parseCotp`.
- `buildCotpData` emits LI `0x02`, type `0xF0`, EOT `0x80`, then the payload; `parseCotp` returns the payload unchanged.
- `parseCotp` returns `null` on a truncated packet, on a length indicator that overruns the buffer, and on a CR whose parameter length overruns (no throw in any case).

- [ ] **Step 3: Implement** `tpkt_cotp.dart` (pure; `dart:typed_data` only). Every function's doc comment states **big-endian**. All parse functions return nullable and never throw.

### Task 2: S7 PDU header + Setup Communication

**Files:**
- Create: `mobile/lib/protocols/s7/s7_pdu.dart`
- Test: `mobile/test/s7_pdu_test.dart`

**Interfaces:**
- Produces: `S7Header { int rosctr; int pduReference; int parameterLength; int dataLength; int errorClass; int errorCode; }`, `S7Message? parseS7(Uint8List)` (header + `parameter` + `data` slices), `Uint8List buildS7({required int rosctr, required int pduReference, required Uint8List parameter, Uint8List? data, int errorClass = 0, int errorCode = 0})`, `SetupComm? parseSetupCommunication(Uint8List parameter)`, `Uint8List buildSetupCommunicationReply({required int maxAmqCalling, required int maxAmqCalled, required int pduLength})`, `int negotiatePduLength(int clientProposal)`, plus ROSCTR/function/area/transport-size/return-code constants.

**Context (all fields BIG-ENDIAN):**
- **Header**: `protocolId 0x32`, `rosctr` u8, `redundancyId` u16 (`0x0000`), `pduReference` u16, `parameterLength` u16, `dataLength` u16 — **plus `errorClass` u8 and `errorCode` u8 present ONLY when `rosctr == 0x03` (`Ack_Data`)**, making that header **12 bytes instead of 10**. Getting this conditional wrong shifts every byte after it.
- ROSCTR: `Job 0x01`, `Ack 0x02`, `Ack_Data 0x03`, `Userdata 0x07`.
- **Setup Communication** parameter (`function 0xF0`): function u8, reserved u8, maxAmqCalling u16, maxAmqCalled u16, **pduLength u16**.
- `negotiatePduLength` returns `min(clientProposal, kS7MaxPduLength)` where `const int kS7MaxPduLength = 480;` — negotiating **down** only, never up. Document that every later response must respect the agreed size.
- Function codes: `Read Var 0x04`, `Write Var 0x05`.
- Area codes: `PE (inputs) 0x81`, `PA (outputs) 0x82`, `MK (merker) 0x83`, `DB 0x84`.
- Item transport sizes: `BIT 0x01`, `BYTE 0x02`, `CHAR 0x03`, `WORD 0x04`, `INT 0x05`, `DWORD 0x06`, `DINT 0x07`, `REAL 0x08`.
- Return codes: `0xFF` success, `0x0A` object does not exist, `0x05` address out of range, `0x03` access denied.

- [ ] **Step 1: Write the failing tests**

Create `mobile/test/s7_pdu_test.dart` covering:
- `parseS7` on a hand-built **Job** (`rosctr 0x01`) reads a **10-byte** header and slices `parameter`/`data` at the right offsets — assert the parsed `pduReference` from two DIFFERENT bytes (e.g. `0x12, 0x34` → `0x1234`) so a little-endian implementation FAILS.
- `parseS7` on a hand-built **`Ack_Data`** (`rosctr 0x03`) reads a **12-byte** header, exposes `errorClass`/`errorCode`, and slices `parameter` starting **2 bytes later** than the Job case. **This pair of tests is the point of this task — write them together.**
- `buildS7` round-trips both, and emits `errorClass`/`errorCode` bytes **only** for `Ack_Data`; assert the emitted total length differs by exactly 2 between the two.
- `parseS7` returns `null` when: the buffer is shorter than the header; `protocolId != 0x32`; or `parameterLength + dataLength` overruns the buffer (no throw).
- `parseSetupCommunication` extracts `pduLength` from a hand-built parameter (assert big-endian).
- `negotiatePduLength`: a client proposing 960 gets 480; proposing 240 gets 240; proposing 0 or a negative-looking value is clamped to a sane floor rather than returning 0 — pick and document the floor.
- `buildSetupCommunicationReply` round-trips through `parseSetupCommunication`.

### Task 4: `S7Map` + area image + Read/Write Var

**The area image is the highest-risk unit in this workstream — it is pure and isolated specifically so it can be attacked hard.**

**Files:**
- Create: `mobile/lib/models/s7_map.dart`, `mobile/lib/protocols/s7/s7_area_image.dart`
- Modify: `mobile/lib/protocols/s7/s7_pdu.dart` (Read/Write Var item parse/build)
- Test: `mobile/test/s7_map_test.dart`, `mobile/test/s7_area_image_test.dart`

**Interfaces:**
- Produces: `S7MapEntry { String tag; String area; int dbNumber; int byteOffset; int bitOffset; String access; }`, `S7Map { List<S7MapEntry> entries; fromJson/toJson; static S7Map autoGenerate(PlcProject); }`; `Uint8List readAreaImage(PlcProject, S7Map, String area, int dbNumber, int startByte, int length)`; `List<S7WriteResult> applyAreaWrite(PlcProject, S7Map, String area, int dbNumber, int startByte, Uint8List data)`; and in `s7_pdu.dart`, `S7Item? parseReadItem(Uint8List)` / item-response builders.

**Context:**
- **Item specification**: `0x12` (variable specification), length-of-following `0x0A`, syntax id `0x10` (S7ANY), transport size u8, count u16, DB number u16, area u8, then a **24-bit (3-byte) address** encoding `byteOffset * 8 + bitOffset`.
- **Response data item**: return code u8, transport size u8, length u16, data, **padded to an even byte count**. **The length field's UNIT depends on transport size** — **bits** for `0x03` (BIT) and `0x04` (BYTE/WORD), **bytes** for `0x09` (octet string). Test both.
- **Write Var**: the parameter carries the item specs; the **data** section carries the write payloads. The **response** is `function 0x05`, item count, then **one return-code byte per item**.
- `S7Map` mirrors `models/modbus_map.dart`. `autoGenerate` packs into `DB1` with natural alignment (2-byte types on even offsets; 4- and 8-byte types on 4-byte boundaries), marks `ioType == 'SimulatedOutput'`, the reserved `System` tag, and `access == 'ReadOnly'` tags as `ReadOnly`, and **skips `STRING`** (an S7 STRING is a struct with max/actual length header bytes — deferred to v2; document that in the file header).
- **Type mapping**: `BOOL`→1 bit, `INT16`→INT 2B, `INT32`→DINT 4B, `INT64`→LINT 8B, `FLOAT64`→**REAL 4B (NARROWING to IEEE-754 single precision — document it)**, `STRING`→skipped.
- **Gap semantics (approved decision):** unmapped bytes inside a requested range **read as `0x00`**, and writes to them are **discarded**.
- A tag only **partially** covered by a write range is **NOT** written (a partial write would corrupt a multi-byte value) and is reported in the result.
- Use `mobile/lib/models/tag_resolver.dart` for reads/writes; do not invent a parallel resolution path.
- **Force-aware:** a write landing on a **forced** or `ReadOnly` tag is refused and the tag left unchanged. Check `root.isForced` **without** any `root.name == tagName` clause — see the Key Facts note; that clause caused a Critical bypass in WS2.

- [ ] **Step 1: Write the failing tests**

`s7_map_test.dart`: `autoGenerate` assigns non-overlapping offsets with correct alignment; marks `SimulatedOutput`/`System`/`ReadOnly` as `ReadOnly`; **skips `STRING`**; `fromJson`/`toJson` round-trip; a JSON entry without `access` defaults to `'ReadWrite'`.

`s7_area_image_test.dart` (against a fixture `PlcProject`):
- Each type encodes at its mapped offset with the correct **width and big-endian byte order** — assert literal expected bytes.
- **`FLOAT64 → REAL` narrowing**: assert `closeTo` with a tolerance near the real float32 error **and** `isNot(equals(original))`, so a non-narrowing implementation FAILS. Choose a value genuinely perturbed by a float32 round-trip (a whole number is not).
- **Bit addressing**: two BOOLs at `byteOffset 0, bitOffset 3` and `bitOffset 5` land in the correct bits of the same byte and do not disturb each other.
- **Gaps read as zero**; a range entirely outside any mapping reads all zeros.
- **Writes to gap bytes are discarded** (no tag changes).
- **A partially covered multi-byte tag is NOT written** and is reported.
- Write to a **`ReadOnly`** entry → refused, tag unchanged.
- **Write to a FORCED tag → refused, tag unchanged — including via a member path beneath a forced root** (e.g. `Tank.Level` with root `Tank` forced). Add the contrast case: a **non-forced** composite's member write SUCCEEDS, proving the refusal is not over-broad.

Plus, in `s7_pdu_test.dart`: item spec parse/build round-trip including the 24-bit address encoding (assert `byteOffset*8 + bitOffset` explicitly); a response data item with transport size `0x04` whose length is in **bits**, and one with `0x09` whose length is in **bytes**; odd-length data padded to even.
